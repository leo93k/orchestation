# 프롬프트 가이드

## 핵심 원칙 (실험 기반)

> "가장 좋은 프롬프트는 프롬프트를 쓰지 않는 것이다"

### 1. PRD > 프롬프트

- PRD(뭘 만들지)가 프롬프트(어떻게 만들지)보다 결과에 훨씬 큰 영향
- Claude는 이미 "어떻게 만드는지" 알고 있음
- 프롬프트에 투자할 시간에 PRD를 다듬어라

### 2. 역할명을 쓰지 마라

- "너는 백엔드 전문가다" → 편향 발생 (프론트엔드 무시)
- "너는 실용주의 개발자다" → 보안 무시, 단일 파일
- 역할명 없이 행동 지침만 주면 품질 향상 (p7 42점 → p8 48점)

### 3. 검증 루프가 품질을 결정

| 검증 횟수 | 평균 점수 |
|----------|---------|
| 0회 | 35/60 |
| 1회 | 48/60 |
| 3~4회 | 50/60 |

- 코드 작성 후 **서버를 띄워서 실행 확인**하는 턴이 중요
- `-p` (print 모드) 대신 에이전트 모드를 사용해야 검증 가능

### 4. 단어 하나가 결과를 바꾼다

| 프롬프트 | 비용 | 점수 |
|---------|-----|------|
| "**잘** 구현해줘" | $0.39 | 50/60 |
| "**간결하게** 구현해줘" | $0.22 | 32/60 |

## 추천 프롬프트

### S Tier (추천)

**"잘 구현해줘"** — 품질 최고, 안정적
- Claude의 기본 판단을 방해하지 않음
- 검증을 알아서 수행

**범용 행동 지침 (역할명 없음)**
```
코드 먼저. 첫 턴부터 파일을 생성하라.
구조는 너의 판단에 맡긴다. 단일 파일 금지.
검증은 실행으로. 브라우저에서 정상 동작하는지 확인하라.
사용자 입력은 반드시 검증하라.
```

### 금지

- "간결하게 구현해줘" → 기능 생략
- 복잡한 역할 설정 → 편향
- 재귀적 오케스트레이션 → 이 규모에서 오버헤드
